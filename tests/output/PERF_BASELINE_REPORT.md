## 四、瓶颈定位

### Top 1: 每日行程填充（40.5%）

- **瓶颈类型**: 大模型串行调用（N 天 × 单日 LLM 生成）
- **根因分析**: 3 天行程需要依次调用 3 次 LLM 逐日生成，
  每次调用包含景点选择、预算拆分、美食/交通/住宿推荐，prompt 长度 2000+ tokens
- **优化方向**: 将 3 天改为并发填充（parallel），
  预计可将该节点耗时压缩至单日最长耗时级别（约降至当前的 35-50%）
- **预期收益**: 总耗时降低 30-40%

### Top 2: 事实校验（19.1%）

- **瓶颈类型**: LLM 校验调用
- **根因分析**: 逐景点逐项校验的 prompt 包含完整行程数据，token 消耗大
- **优化方向**: 规则兜底优先（本地正则/字典匹配），仅在规则无法判定时回调 LLM
- **预期收益**: 总耗时降低 5-10%

### Top 3: 景点检索+框架生成（6.0%）

- **瓶颈类型**: 工具调用 + LLM 框架生成双阶段串行
- **根因分析**: 内部包含 spot_retriever 工具调用（LLM 检索景点）
  和 LLM 框架生成两次独立的大模型交互，串行累加
- **优化方向**: 
  1. 为热门城市预置景点缓存（本地 JSON/ChromaDB），跳过 LLM 检索
  2. spot_retriever 和框架生成之间减少无效等待
- **预期收益**: 热门城市该节点可降至 10-15s，总耗时降低 10-15%
